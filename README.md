# ГОРОД — государственный экономический реестр

Vite + React проект, готовый к деплою на Vercel. Общий игровой мир хранится
в Supabase (Postgres), поэтому все, кто открывают сайт, действительно играют
в одном мире — как и задумывалось в оригинальном файле.

## 1. Локальный запуск

```bash
npm install
cp .env.example .env.local   # впишите свои ключи Supabase (см. ниже)
npm run dev
```

## 2. Настройка Supabase (один раз)

1. Зайдите на https://supabase.com → создайте новый проект (бесплатный тариф подходит).
2. В проекте откройте **SQL Editor** → **New query**, вставьте содержимое
   файла `supabase.sql` из этого репозитория и нажмите **Run**.
   Это создаст таблицу `kv_store` (общее хранилище всего игрового мира)
   и разрешит анонимному ключу читать/писать в неё.
3. Откройте **Project Settings → API**. Вам нужны два значения:
   - **Project URL** → это `VITE_SUPABASE_URL`
   - **anon public key** → это `VITE_SUPABASE_ANON_KEY`

⚠️ Используется именно `anon` ключ (публичный, для браузера), а не `service_role`
(его никогда нельзя класть во фронтенд). Доступ к таблице открыт всем посетителям
сайта намеренно — это и есть общий мир без отдельного бэкенда.

## 3. Деплой на Vercel

1. Загрузите этот проект в GitHub-репозиторий (или используйте Vercel CLI: `vercel`).
2. На https://vercel.com → **New Project** → выберите репозиторий.
   Vercel сам определит, что это Vite-проект (build command `vite build`,
   output directory `dist`) — ничего менять не нужно.
3. В настройках проекта Vercel → **Settings → Environment Variables** добавьте:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
   (те же значения, что и в `.env.local`)
4. Нажмите **Deploy**.

После этого при каждом коммите в основную ветку Vercel будет пересобирать
и обновлять сайт автоматически.

## Важно про безопасность

- Пароль администратора (`ADMIN_NICK` / `ADMIN_PASSWORD` в `src/App.jsx`)
  и все пароли игроков хранятся и проверяются **на клиенте, в открытом виде**
  в общей таблице `kv_store`. Это ровно то же поведение, что было в
  оригинальном файле с `window.storage` — architecturally ничего не изменилось,
  просто хранилище теперь настоящее. Если это игра "для своих" — не критично.
  Если планируете публичный проект с реальными ставками — стоит вынести
  авторизацию и приватную логику на сервер (например, Supabase Auth +
  Edge Functions / Row Level Security по пользователю), это уже отдельная
  задача архитектуры, а не просто сборка проекта.
- Таблица `kv_store` открыта на чтение и запись всем анонимным посетителям —
  это осознанный компромисс ради простоты (без сервера/бэкенда). Любой,
  кто знает `anon`-ключ (он всё равно виден в собранном JS-бандле), может
  напрямую читать/менять данные в обход интерфейса. Для хобби-проекта
  и игры с друзьями это обычно приемлемо.

## Структура проекта

```
├── index.html
├── package.json
├── vite.config.js
├── tailwind.config.js
├── postcss.config.js
├── supabase.sql          ← SQL-схема, выполнить один раз в Supabase
├── .env.example
└── src/
    ├── main.jsx
    ├── index.css
    ├── App.jsx            ← вся игровая логика (ваш исходный файл)
    └── lib/
        └── storage.js     ← замена window.storage на Supabase
```
